# TIX-019: Universe-Wide Player Records

Status: Draft mediation ticket
Priority: High
Target project: `BadmintonManager`
Target screens: Player Profile `Career` tab, player profile head-to-head sections, tournament archives
Prepared on: 2026-05-19
Primary files: `app/pages/PlayerProfilePage.tsx`, `game/selectors/player.ts`, `game/career/models.ts`, `game/career/events.ts`, `game/tournament/tournament.ts`, `game/store/save.ts`, `tests/unit/player-profile.test.ts`, `tests/unit/career.test.ts`, `e2e/app.spec.ts`
Reference input: `docs/scratchpad.md` line 71
Related archived tickets: `docs/arc_tix/TIX-013-player-career-history-and-head-to-head.md`, `docs/arc_tix/TIX-009-universal-player-addressing-system.md`, `docs/arc_tix/TIX-008-universal-tournament-home-system.md`
Related active ticket: `docs/active_tix/TIX-017-parallel-universe-tournament-simulation.md`

## 1. Commander Intent

Make player career profiles reflect the whole save universe, not only matches involving the managed player.

Every athlete should have a career record built from all matches recorded in the simulation, including matches generated by tournament quick simulation. The managed athlete remains important, but they should not be the only source of truth.

The profile rule is:

**Player Profile Record = all recorded matches for that player in the career universe**

## 2. Mediation Context

`TIX-013` added the right display goal: all-time W-L, win percentage, titles, runner-up finishes, and head-to-head records.

This mediation ticket sharpens the data interpretation. Those records should not be calculated from a managed-player-centric slice. They should be calculated across the full simulated universe established by `TIX-017`.

## 3. Problem Statement

The current profile direction risks answering only:

**How did this player relate to my managed athlete?**

That is useful, but incomplete. A living sports world also needs to answer:

**What has this player done in the universe?**

Players should gain wins, losses, finals, titles, and head-to-head histories through all tournament matches that the save records.

## 4. Career Record Contract

For every player profile, calculate:

- all-time wins,
- all-time losses,
- win percentage,
- titles,
- runner-up finishes,
- finals reached,
- best result by tier if practical,
- most-played opponents,
- head-to-head records against any opponent with recorded matches.

The record source should include:

- manually played managed matches,
- quick-simulated non-managed matches,
- archived/imported match records if introduced for migration,
- tournament final outcomes written at event closeout.

Do not include:

- projected matches,
- unplayed future fixtures,
- bracket possibilities,
- speculative head-to-heads,
- old-save history that was never recorded.

## 5. Head-To-Head Contract

Head-to-head should be universe-wide.

For a given player pair:

**H2H(A, B) = all recorded matches where A played B**

This includes matches where neither athlete is the managed player.

When viewing a non-managed player, the profile may still highlight their record against the managed athlete, but that should be a special spotlight, not the only head-to-head model.

Suggested sections:

| Section | Purpose |
| --- | --- |
| Career Record | Full universe W-L and win percentage. |
| Achievements | Titles, runner-up finishes, finals, best event results. |
| Head-To-Head Leaders | Most-played opponents across the universe. |
| Vs Managed Player | Optional spotlight when matches exist against the managed athlete. |

## 6. Data Dependency

This ticket depends on `TIX-017` for complete universe match records.

If `TIX-019` is implemented first, it should still build selectors that are ready for full-universe records, but empty states must remain honest until the data exists.

The selector should be shaped around records, not managed-player assumptions:

```ts
type PlayerCareerProfileRecord = {
  playerId: string;
  wins: number;
  losses: number;
  winPercentage: number | null;
  titles: number;
  runnerUps: number;
  finals: number;
  headToHeads: Array<{
    opponentId: string;
    played: number;
    wins: number;
    losses: number;
    winPercentage: number;
  }>;
};
```

## 7. UI Contract

The Player Profile `Career` tab should make the universe scope clear through content, not long explanation text.

Required display:

- `W-L`,
- `Win %`,
- `Titles`,
- `Runner-up`,
- compact achievement list,
- top head-to-head table,
- player-profile links for opponents,
- honest empty state for players with no recorded universe matches.

Optional display:

- `Vs Managed Player` spotlight,
- recent form from last five recorded matches,
- best tournament result by tier.

## 8. Absolute Rules

- Do not calculate records only from matches involving the managed athlete.
- Do not fabricate records for players with no recorded match history.
- Do not count possible future knockout rounds.
- Do not double-count a match if it exists both in a bracket snapshot and match history.
- Do not break old saves that lack universe-wide records.
- Do not remove managed-player-specific context; place it as a spotlight inside the broader universe record.

## 9. Acceptance Criteria

- [ ] Player career records are calculated from all recorded save-universe matches.
- [ ] Non-managed players can show W-L records from matches not involving the managed athlete.
- [ ] Head-to-head tables include any recorded opponent pair, not only pairings against the managed athlete.
- [ ] Optional managed-player head-to-head spotlight appears only when relevant records exist.
- [ ] Titles and runner-up finishes reflect completed tournament outcomes.
- [ ] Old saves with no match history show honest empty states.
- [ ] Selectors avoid double-counting archived bracket snapshots and match records.
- [ ] Opponent names use universal player addressing and open player profiles.

## 11. Definition Of Done

Player profiles feel like records from a shared badminton universe:

**Every player → their own career history**

The manager's athlete is central to the user's story, but not central to the world's data model.
