# V0.2.3 Game-algorithm

Status: Active
Owner: BinaryOutlook
Last updated: 2026-05-05
Iteration type: Match simulation fidelity and two-tier game algorithm system

## 1. Summary

`v0.2.3` is the game-algorithm release.

Its job is to turn the match simulation fidelity plan into the active implementation target:

```text
active managed match -> high-fidelity detailed simulator
background tournament match -> quick calibrated simulator
```

The player-facing goal is not more hidden complexity for its own sake. The goal is a match engine
that feels more like real badminton because tactics, fatigue, pressure, shot patterns, and bounded
luck create outcomes the coach can understand.

## 2. Release Thesis

The core loop remains:

$$
\text{select athlete} \rightarrow \text{scout opponent} \rightarrow \text{choose tactic}
\rightarrow \text{manage live match} \rightarrow \text{review run}
$$

`v0.2.3` strengthens the middle of that loop.

The active match should feel richer because the engine tracks more of the state that makes
badminton legible:

- initiative
- contact quality
- target intent
- movement stress
- line judgment
- retrieval quality
- tactical pressure
- fatigue arcs
- score pressure

Unseen tournament matches should stay fast by using a quick simulator that preserves macro
outcomes without generating fake shot-by-shot detail.

## 4. Product Rules

This release must preserve:

- deterministic outputs for identical seeds, players, tactics, choices, and fidelity mode
- simulation logic outside React components
- local-first operation
- the 16-player knockout tournament shape
- the player fantasy of coach, not athlete
- commentary and telemetry that explain engine state rather than inventing decorative causes

It must not add:

- direct racket controls
- a backend
- licensed athlete likenesses
- exact shuttle physics
- 3D broadcast simulation

The target is:

$$
\text{believable badminton}
=
\text{ratings}
+ \text{tactics}
+ \text{pressure}
+ \text{fatigue}
+ \text{bounded variance}
$$

## 5. Implementation Scope

### 5.1 Simulation fidelity boundary

Introduce an explicit fidelity selector:

```ts
type SimulationFidelity = "detailed" | "quick";
```

Managed matches should use `detailed`.

Non-managed tournament matches should use `quick`.

Both paths should return a compatible `MatchResult` so tournament advancement, recap surfaces, and
tests can stay coherent.

Initial implementation status:

- `SimulationFidelity` is now modeled as `"detailed" | "quick"`.
- `simulateMatch()` remains the detailed path for backwards compatibility.
- `simulateMatchByFidelity()` dispatches to detailed or quick simulation.
- Tournament matches record the simulation fidelity used to resolve them.

### 5.2 Detailed simulator

The current seeded rally engine should become the detailed simulator path.

The first implementation step should preserve existing behavior while making the boundary explicit.
After the split is stable, deepen the detailed path with the model described in
`docs/reference/match-simulation-fidelity.md`.

Priority additions:

- explicit rally state
- richer shot intent categories
- target intent separate from court zone
- quality bands
- contextual error families
- line judgment outcomes
- retrieval quality bands
- movement-load fatigue
- score-pressure penalties
- short-lived directive effects

### 5.3 Quick simulator

The quick simulator should not be a pure match coin flip.

Use quick point simulation:

```text
derive point win probability
simulate points directly
apply light fatigue and pressure state
stop at 21 by 2 or 30 cap
repeat until one side wins two sets
generate aggregate stats and summary events
```

The core point probability should use a logistic shape:

$$
P(A \text{ wins point}) =
\sigma\left(\frac{X}{K}\right)
$$

Where \(X\) blends player strength, tactic fit, style matchup, fatigue edge, pressure edge, and
seeded variance.

Initial implementation status:

- quick mode simulates points directly instead of shot loops
- quick mode preserves best-of-three, rally scoring to `21`, win by `2`, and the `30` point cap
- quick mode generates lightweight point summaries, aggregate stats, and summary events
- non-managed tournament matches now resolve through quick mode

### 5.4 Two-tier display system

The active match display should show detailed truth:

- point feed
- tactical directives
- between-set talks
- stamina and momentum
- key point explanations
- structured commentary from engine events

Background match display should show compressed truth:

- winner
- scoreline
- upset or decider tags
- broad match style summary
- aggregate stats when useful

Background matches should not display fake shot-by-shot commentary.

Initial implementation status:

- active managed matches retain detailed sessions, point feed, directives, and team talks
- background matches store compressed summary events
- completed background matches render a compact bracket context line from those summary events

## 6. Verification Expectations

Keep these commands green:

- `npm run typecheck`
- `npm run test`
- `npm run build`
- `npm run test:e2e`

Algorithm-specific tests should cover:

- detailed mode remains deterministic
- quick mode is deterministic
- managed tournament matches use detailed mode
- non-managed tournament matches use quick mode
- quick mode obeys best-of-three scoring, `21` by `2`, and the `30` point cap
- quick mode returns a valid tournament-compatible result
- detailed mode still supports live directives and between-set talks
- stronger players win most seeded batches in both modes
- quick and detailed mode win rates stay within accepted calibration bands
- aggregate stats remain plausible and internally consistent

## 7. Calibration Targets

Balance should be tested through seeded batch simulation.

Initial calibration questions:

- How often does an elite athlete beat a baseline tour athlete?
- How often can a lower-rated tactical fit upset a higher-rated opponent?
- Does high risk increase both winners and errors?
- Does fatigue matter more in third sets than first sets?
- Does quick mode preserve detailed-mode winner rates closely enough for background matches?

Start with this parity target for important matchup bands:

$$
\left|P_{\text{detailed}}(\text{A wins}) - P_{\text{quick}}(\text{A wins})\right| \le 0.05
$$

This target can loosen temporarily while the first quick simulator is introduced, then tighten once
batch tooling exists.
