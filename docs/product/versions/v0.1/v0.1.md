Status: Draft
Owner: BinaryOutlook

Iteration type: Foundation, headless engine, and tournament MVP

## 1. Summary

`V0.1` establishes the first playable version of **Badminton Manager**.

This iteration is not trying to build the entire management game.

It is trying to prove three things:

- the technical stack is lightweight and maintainable
- the singles match engine can produce believable results
- a full 16-player tournament can be completed through a browser interface

This iteration should:

- scaffold the local-first React and Vite app
- define the player, tactic, and save schemas
- implement deterministic rally, set, and match simulation
- seed a 16-player tournament
- expose match flow through a simple commentary-first interface

This is a foundation and gameplay iteration. It is not a late-stage polish pass.

## 2. Problem Statement

The project currently has a product thesis and documentation, but no runnable game.

The main danger is building too much around an unproven simulation.

If `v0.1` jumps into:

- large UI systems
- long-term progression
- backend setup
- fancy court presentation

before the core engine is trustworthy, the project will spend effort in the wrong place.

`V0.1` must make the match engine real first and keep the browser shell thin.

## 3. Goals

This iteration should:

- initialize the chosen Option A stack
- create a pure simulation boundary separate from React
- implement seeded singles match resolution
- make tactics, stamina, and pressure visibly matter in match outcomes
- support a full 16-player knockout tournament run
- present scores and commentary clearly enough to debug and play
- establish save versioning for the local-first run state

## 4. Non-Goals

This iteration does not:

- implement doubles
- add a long-term career mode
- require a backend or database
- add 2D court animation
- add deep training, finances, or recruitment systems
- optimize balance beyond obvious glaring problems

## 6. Architecture Requirements

### 6.1 Simulation boundary

Requirements:

- the match engine must remain separate from React components
- engine modules must accept explicit inputs and return explicit outputs
- engine modules must be testable without browser APIs

### 6.2 Deterministic RNG

Requirements:

- match outcomes must be reproducible for a given seed and identical inputs
- randomness consumption order must stay stable enough for regression testing

### 6.3 Save state

Requirements:

- save payloads must include a version field
- saves must be validated before hydration
- corrupt save handling must fail safely

## 7. Gameplay Requirements

### 7.1 Match engine

Requirements:

- best-of-three match structure
- rally scoring to `21`, win by `2`, cap at `30`
- player ratings must affect shot choice and point outcomes
- tactics must influence behavior without replacing player identity
- fatigue and pressure must affect late-match outcomes

### 7.2 Tournament flow

Requirements:

- the player can enter a 16-player bracket
- the player can complete the entire bracket if they keep winning
- bracket progression remains readable between rounds

### 7.3 Presentation

Requirements:

- the UI must show score, set count, and commentary
- the player must be able to choose tactics before a match
- the player must receive at least one between-set intervention option

## 8. UX Acceptance Criteria

The iteration is successful when all of the following are true:

- the app can be started locally with the documented setup
- the player can complete at least one tournament run from opening bracket to champion screen
- the same seed with the same players and tactics produces the same match result
- commentary and point summaries make it understandable why the match turned
- strong players usually beat weaker players, but upsets remain possible
- the UI feels like a coaching simulation rather than an action game shell

## 9. QA Notes

Check these scenarios before closing the iteration:

- a heavy favorite against a weak opponent
- two evenly matched players under different tactic packages
- a three-set match with visible stamina decline
- a close set that reaches pressure states above `18-18`
- save and reload in the middle of a tournament

## 10. Open Questions

- whether the first version should expose one or multiple managed athletes is still open, but current planning assumes one
- whether the MVP should display detailed rally-by-rally stats or only a compact post-match summary remains open
- whether between-set talks should be optional or mandatory depends on the pacing of the first playable build
