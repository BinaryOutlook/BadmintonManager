# V0.2 PRD

Status: Draft
Owner: BinaryOutlook
Last updated: 2026-04-28
Iteration type: Super build, command-center redesign, and tactical operations expansion

## 1. Summary

`v0.2` is the first premium-feeling release target for **Badminton Manager**.

`v0.1` proved that the project can ship a deterministic singles engine, a 16-player knockout flow, and a playable browser loop.

`v0.2` must prove something different:

- the product can feel like a serious badminton management game instead of a prototype
- the interface can explain the match rather than merely display it
- the player can read, influence, and review a match through a richer coaching console
- the mockups in `docs/product/versions/v0.2/UIDEMOs` can be translated into both UI and real feature scope

This version is not only a visual refresh.

It is a substantial gameplay presentation upgrade that adds scouting, richer athlete dossiers, expanded tactic framing, point-by-point live match operations, telemetry, and a more satisfying tournament recap.

## 2. Assumptions And Source Inputs

### 2.1 Assumptions

Unless corrected later, this PRD assumes:

1. `v0.2` remains a local-first `React + TypeScript + Vite` web app.
2. The player fantasy remains coach and manager, not direct athlete control.
3. The UI mockups define both visual direction and functional intent.
4. `v0.2` is still singles-only and still built around one tournament run at a time.
5. The "active roster" shown in the mockups is a run-start selection surface, not a full long-term squad management system.
6. Mockup athlete names, faces, and tournament branding are illustrative unless a separate content and legal decision says otherwise.

## 3. Problem Statement

`v0.1` is playable, but it still feels like an early debug shell.

Current weaknesses:

- the setup screen is informative but not memorable
- the bracket screen lacks opponent scouting and event identity
- the live match screen is readable, but too coarse for the command-center vision
- match drama is muted because the player mostly watches set chunks rather than reading live swings
- recap quality is too light for a "super build" release

The product risk for `v0.2` is no longer "does the engine work at all?"

The product risk is now "can the player feel like a badminton strategist operating a serious match desk?"

## 4. Release Thesis

`v0.2` should feel like entering an elite tournament operations room.

Every screen should strengthen at least one of these outcomes:

- confidence in athlete selection
- clarity in opponent preparation
- tension during live match management
- meaning behind tactical interventions
- satisfaction at the end of a run

If a feature looks impressive but does not improve one of those outcomes, it is not core `v0.2` work.

## 5. Goals

This iteration should:

- deliver the new command-center UI direction shown in the mockups
- preserve the deterministic simulation boundary from `v0.1`
- upgrade the live match flow from set-by-set presentation to point-by-point presentation
- add opponent scouting and matchup framing before managed matches
- add richer athlete identity, dossier, and form surfaces
- expand tactic presentation from simple labels to intent-driven coaching stances
- introduce live directives that let the player influence short-term tactical emphasis without becoming an action game
- add a stronger tournament recap with run-level telemetry and narrative summary
- make the entire run feel cohesive through a persistent shell and shared product identity

## 6. Non-Goals

This iteration does not:

- add doubles
- add a long-term career or calendar mode
- add real-time reflex controls
- add a backend, auth, or cloud save dependency
- add licensed athlete likeness as a requirement
- add a full searchable database of athletes, tournaments, or historical stats
- add 2D or 3D rally animation
- turn shell icons like notifications or monitoring into full live-service systems

## 7. Core Player Experience Loop

The `v0.2` run loop should be:

1. Enter the tournament deployment screen.
2. Browse the available athletes and select the managed athlete for this run.
3. Review dossier information and choose the opening strategic override.
4. Enter the tournament command center and inspect event context, the next opponent, threat level, and bracket path.
5. Lock a match tactic and start the managed match.
6. Read the live tactical feed point by point, use directives during live play, and use a stronger team-talk intervention at set breaks.
7. Advance through the bracket until eliminated or crowned champion.
8. Review a recap screen that explains the path, turning points, and aggregate telemetry for the run.
9. Start a fresh session from a clear reset point.

## 8. Release Scope

### 8.1 Global app shell

`v0.2` must introduce a persistent shell that frames the whole experience.

Requirements:

- top navigation and side navigation with a clear active state
- stable product identity for the game across all screens
- persistent entry point for `TACTICAL_INTEL`
- visible phase or mode context such as `LIVE`, `SQUAD`, or `BRACKETS`
- support for current user or coach identity framing
- consistent dark command-center styling across all screens

Clarification:

- search, notifications, monitoring, and settings may appear as shell affordances in `v0.2`
- they do not need deep underlying systems unless explicitly scoped later

### 8.2 Athlete tactical briefing screen

This screen replaces the current lightweight setup view with a more ceremonial "tournament deployment" experience.

Functional requirements:

- athlete roster browsing with visibly selectable athlete cards
- selected athlete dossier with identity, country, ranking or seed, and portrait slot
- derived capability bars for at least `Power`, `Speed`, and `Stamina`
- concise form summary that helps the player understand the athlete's current competitive profile
- opening tactic selection presented as strategic stances rather than plain toggles
- clear primary CTA to start the tournament

Design intent:

- the player should feel like they are selecting an operative for a major event
- the chosen athlete should feel more important than a list row
- tactic choice should feel like committing to a coaching plan

Scope note:

- this is still one managed athlete per run
- it is not yet full squad progression, contract management, or season roster management

### 8.3 Tournament command center

This screen is the pre-match preparation desk for the current managed round.

Functional requirements:

- named event presentation with tournament title, category, and round
- optional surface for prize pool or prestige metadata
- next opponent panel with threat level and matchup summary
- opponent strength bars or badges for major qualities such as power, stamina, and net play
- tactic lock-in panel with at least three distinct tactical options
- visible summary of the managed athlete entering the match
- bracket visualization with completed matches, live managed match, and future placeholders
- clear CTA to enter the managed match

Behavioral requirements:

- non-managed matches remain auto-resolved by the simulation
- the current managed match must stand out visually in the bracket
- the player must understand where they are in the tournament within a few seconds

### 8.4 Live match command center

This is the most important change in `v0.2`.

The live match screen must move from coarse set-level pacing to granular point-level pacing while still remaining a management game.

Functional requirements:

- live scoreboard with current point score, current set, set lead, and serving indicator
- visible court and event context
- point-by-point progression controlled by a `Simulate Next Point` action
- live tactical feed that logs point outcomes, rally warnings, and tactical alerts
- telemetry panels for both competitors
- readable stamina display for both competitors
- readable momentum display for both competitors
- visible count or summary for match mistakes such as errors
- at least one additional headline telemetry stat per side, such as smash speed, pressure rating, or attack success
- directive buttons that let the player queue short-term tactical emphasis during live play
- between-set team-talk surface that remains stronger than normal directives
- clear match-complete state with transition back into bracket flow

Interaction model:

- directives are short-horizon nudges, not direct shot controls
- team talks remain higher-impact interval decisions at set breaks
- the player never manually places a shuttle or times a swing

Required event categories in the live feed:

- point result
- long rally or fatigue warning
- tactical alert or pattern detection
- set end summary
- directive applied confirmation

### 8.5 Tactical Intel layer

The recurring `TACTICAL_INTEL` affordance in the mockups should become a real contextual explanation layer.

This layer can ship as a drawer, side panel, or modal.

It should provide:

- athlete strengths and fit with the chosen tactic
- opponent threat explanation and matchup notes
- meaning of telemetry labels such as momentum or pressure
- rationale behind surfaced tactical alerts
- recap explanation for why a run succeeded or failed

This feature is important because `v0.2` is trying to make systems legible, not just stylish.

### 8.6 Tournament recap and results

The end-of-run screen must feel conclusive and earned.

Functional requirements:

- championship and elimination variants
- result headline and short narrative summary
- bracket path review with actual opponents and match results
- run telemetry panel
- clear CTA to start a new session

Run telemetry should include at least:

- matches won
- aggregate set or point performance
- total attacking output such as smashes or winners
- stamina drain or endurance cost across the run
- one high-drama metric such as longest rally, peak exertion, or pressure survival

## 9. System Expansion Required For V0.2

### 9.1 Point-by-point match session model

The current match presentation resolves one set at a time.

That is not enough for the live command center shown in the mockups.

`v0.2` requires:

- current set point score in live state
- point history emitted incrementally
- event timestamps or match-clock-style ordering for feed presentation
- support for pausing between points for player directives
- continued deterministic output for the same seed and same choices

### 9.2 Expanded tactics model

The tactic system should evolve from four simple presets toward a clearer coaching language.

`v0.2` should support:

- pre-match tactic presets with strong identity and readable trade-offs
- live directives that temporarily bias tactical behavior
- set-break team talks that remain distinct from live directives
- visible explanation of risk, tempo, and intent

The mockups imply tactic families such as:

- balanced control
- aggressive smash or fast attacks
- spread court
- defensive wall
- defensive net-play

Exact final naming can change, but the tactical variety and readability should not shrink below the mockup intent.

### 9.3 Scouting and threat model

The bracket desk mockups imply a real scouting layer.

`v0.2` should add:

- opponent threat level such as `Low`, `Medium`, or `High`
- compact opponent strengths summary
- matchup notes derived from player attributes and tactic fit
- clear explanation of why a specific opponent is dangerous

This layer should be derived from real player data and deterministic matchup rules, not hand-written flavor only.

### 9.4 Telemetry and derived metrics

The mockups introduce a more sports-science presentation layer.

`v0.2` should formalize which metrics are:

- direct engine outputs
- deterministic derived values
- presentation-only labels

Requirements:

- no major telemetry label should be fake or arbitrary
- capability bars must map to understandable formulas from player ratings
- run recap numbers must be aggregated from actual match data
- if a metric is synthetic, its rule must still be deterministic and documented

Candidate telemetry types:

- power
- speed
- stamina
- momentum
- errors
- smash output
- longest rally
- stamina drain
- peak exertion

### 9.5 Run aggregation and recap logic

The game currently resolves matches and advances the bracket, but the end state is light.

`v0.2` requires:

- per-match stat capture suitable for a later recap
- run-level aggregation
- generated narrative beats based on actual turning points
- recap layout that can scale if more stats arrive in future versions

### 9.6 Persistence and save migration

The live session model will become richer in `v0.2`.

Requirements:

- save payload version must be bumped when the shape changes
- old saves must fail safely or migrate intentionally
- in-progress live matches must persist point state, directives, and set context correctly

## 12. Code Style

The style rule for `v0.2` is simple:

the UI presents state and sends intent, while the engine resolves consequences.

Example:

```ts
const session = simulateNextPoint(state.liveMatch.session, chosenDirective);

set({
  liveMatch: {
    ...state.liveMatch,
    session
  }
});
```

Conventions:

- keep simulation logic in `src/game/core/`
- derive view-model style values in helpers or the store, not inside bulky JSX
- use reusable theme tokens instead of one-off colors for the redesign
- name tactics, directives, and telemetry labels for coaching readability first

## 13. Testing Strategy

### 13.1 Unit tests

Use `Vitest` to cover:

- point-by-point determinism
- directive effects on short-horizon behavior
- team-talk effects at set boundaries
- scouting derivation
- telemetry formulas
- recap aggregation

### 13.2 Integration tests

Use `Vitest` to cover:

- setup selection to tournament creation
- tactic lock-in to live match initialization
- point progression and directive application
- team-talk flow between sets
- bracket advancement after live match completion
- save and restore during a live match

### 13.3 E2E tests

Use `Playwright` to cover:

- full run start from the deployment screen
- pre-match scouting and tactic lock-in
- at least one live match using point-by-point progression
- end-of-run recap rendering
- reset into a fresh session

## 14. Boundaries

### 14.1 Always do

- preserve deterministic outputs for identical seeds and choices
- keep outcome logic outside React components
- ensure telemetry is backed by documented formulas or real event aggregation
- make the new shell responsive on desktop and usable on narrower widths
- preserve local-first play with safe save handling

### 14.2 Ask first

- adding a new major dependency or UI framework
- changing the project to a routed multi-page architecture
- introducing backend features
- replacing the fictional roster with real athlete content
- adding features that imply season mode, squad persistence, or licensing concerns

### 14.3 Never do

- turn the live match into reflex gameplay
- decide rally or point outcomes inside UI components
- ship fake headline stats with no deterministic rule behind them
- let shell affordances like notifications become blockers for the core tournament loop
- sacrifice readability for visual spectacle

## 15. Success Criteria

`v0.2` is successful when all of the following are true:

- the app feels like a coherent command-center product rather than a prototype shell
- the player can complete a full tournament run through the new four-screen flow
- the live match can progress point by point while remaining deterministic
- directives and team talks visibly affect how the match unfolds
- opponent threat and tactic choice are understandable before match start
- recap output explains the path and turning points of the run
- the same seed and same decisions produce the same run outcome and feed
- the redesign leaves room for future systems instead of boxing the game into a dead-end layout

## 16. QA Notes

Check these scenarios before calling the iteration complete:

- favorite versus underdog with a clear threat mismatch
- two balanced players with different tactical stances
- a live match where momentum swings after a directive
- a set break where a team talk changes the third-set shape
- a long rally sequence that creates visible fatigue in the feed
- save and reload in the middle of a live point-by-point match
- elimination recap versus championship recap

## 17. Open Questions

- Should `v0.2` keep the current fictional roster or introduce a new branded content set?

- How many live directives should be available at once before the experience starts feeling too busy?
- Should `TACTICAL_INTEL` be available as a universal drawer or change form screen by screen?
- Should shell elements like search and notifications remain placeholders in `v0.2`, or do we want a minimal functional version of one of them?
