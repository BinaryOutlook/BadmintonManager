# V0.2.4 UI Framework And Season-Ready Layout

Status: Planned / follows `v0.2.3`
Owner: BinaryOutlook
Last updated: 2026-05-06
Iteration type: UI layout framework, feature-page separation, and future season preparation

## 1. Summary

`v0.2.4` is the interface-structure release.

The goal is to clean up the current command-center layout before larger game systems arrive. Previous
versions proved the core loop, expanded the roster, and deepened match simulation. The next risk is
different: as more content enters the game, the interface can become a crowded stack of panels unless
every feature has a clear home, purpose, and interaction shape.

This version introduces a stronger UI framework inside the existing local-first React app:

```text
app shell -> feature pages -> focused overlays -> engine-backed actions
```

The release should prepare the interface for:

- a larger selection of available games or events, such as `32` selectable competitions
- a multi-game season structure where the player can progress across several events
- a calendar system that gives those events, matches, and rest windows a readable timeline

`v0.2.4` does not need to complete the full season simulation. Its job is to make the interface ready
for that future work without burying the existing tournament loop.

## 2. Release Thesis

The product is moving from a single compact tournament console toward a broader manager hub.

The UI must therefore stop behaving like one large screen that happens to contain everything. It
should become a set of purposeful workspaces:

$$
\text{Strong Feature UI} = \text{Decision} + \text{Context} + \text{Action} + \text{Feedback}
$$

Each page should answer one plain question:

> What is the coach trying to decide here?

If a page cannot answer that, the feature is not ready to become a page.

## 3. Why This Version Exists

`v0.2`, `v0.2.1`, `v0.2.2`, and `v0.2.3` made the game more capable, but they also increased the
amount of information competing for attention:

- roster selection has more athletes
- the bracket has more background-match context
- live matches now expose richer point-by-point state
- scouting, tactics, telemetry, and tactical intel all want screen space
- upcoming season and calendar features will add many more planning surfaces

Without a UI framework, future versions will be forced to squeeze new systems into whatever panel is
nearest. That would weaken the coach fantasy because the player would spend more energy finding
controls than making strategic decisions.

`v0.2.4` is the cleanup and preparation pass that prevents that drift.

## 4. Product Rules

This release must preserve:

- local-first operation
- the existing `React + TypeScript + Vite + Zustand` direction
- simulation logic outside React components
- deterministic match and tournament outputs for identical seeds and choices
- the coach-not-athlete product rule
- the current 16-player tournament loop as the playable core
- commentary, telemetry, and recaps that explain real engine state

This release must not add:

- direct racket controls
- a backend, auth, cloud save, or database dependency
- licensed athlete likenesses
- full season progression rules before their model is designed
- a replacement UI framework or state library without an explicit decision

Important wording:

- "UI framework" in `v0.2.4` means local app structure, reusable layout primitives, navigation
  contracts, and overlay patterns.
- It does not mean replacing React or adopting a large external design system by default.

## 5. UI Architecture Direction

### 5.1 Page framework

The app should keep the single-page web architecture, but behave as a multi-page product through a
clear internal page model.

Recommended shape:

```text
CommandShell
  |
  +-- PrimaryNavigation
  +-- SidebarContext
  +-- PageCanvas
  |     |
  |     +-- SetupPage
  |     +-- CommandHomePage
  |     +-- SquadPage
  |     +-- GamesPage
  |     +-- SeasonPage
  |     +-- CalendarPage
  |     +-- BracketPage
  |     +-- LiveMatchPage
  |     +-- ReviewPage
  |
  +-- OverlayHost
        |
        +-- TacticalIntelOverlay
        +-- AthleteDossierOverlay
        +-- OpponentScoutOverlay
        +-- EventDetailsOverlay
        +-- MatchSummaryOverlay
        +-- ConfirmActionOverlay
```

The first implementation can use local app state for page selection. A URL router is optional and
should only be introduced if it clearly improves deep-linking, browser history, or testing.

### 5.2 Page purpose contract

Every major page should have a small contract before implementation.

| Page | Player decision | Primary context | Primary action |
| --- | --- | --- | --- |
| Setup | Which athlete and opening tactic starts the run? | roster, athlete identity, tactic stance | start tournament |
| Command Home | What needs attention next? | current run state, next match, alerts | continue to next task |
| Squad | Who is in the program or available pool? | athlete list, dossiers, ratings | inspect or select athlete |
| Games | Which competition should the coach enter? | available games/events, prestige, draw size, status | choose or preview event |
| Season | How is the multi-game campaign progressing? | season record, points, event path | advance to next scheduled item |
| Calendar | When do events and matches happen? | dates, event windows, rest windows | inspect day or jump to event |
| Bracket | Who is next and what is the path? | bracket, opponent scout, tactic lock | enter managed match |
| Live Match | How should the coach adapt now? | score, stamina, point feed, directives | simulate point or apply decision |
| Review | What did the run teach? | results, telemetry, turning points | start next run or return home |

The design rule is:

```text
one page -> one dominant player intent
```

Secondary information can exist, but it should support the dominant intent instead of competing
with it.

### 5.3 Overlay and pop-up system

Pop-up windows should become a deliberate interaction layer, not ad hoc panels.

Use overlays for information or choices that are important but temporary:

- athlete dossier
- opponent scouting detail
- event or game details
- tactic explanation
- match summary
- reset confirmation
- tactical intel

Overlay rules:

- overlays must have a named type and a single owner in app state
- closing an overlay must never change simulation results
- destructive actions must require confirmation
- overlays should support keyboard dismissal where appropriate
- only one modal-level overlay should be active at a time
- overlays may read engine or store state, but must not resolve gameplay outcomes

### 5.4 Navigation rules

Navigation should communicate real product structure.

- Top navigation selects broad product areas, such as `Home`, `Squad`, `Games`, `Season`, and
  `Calendar`.
- The sidebar should show context-sensitive actions for the active page rather than a second,
  competing page tree.
- The live match and bracket paths should remain prominent because they are the playable core.
- Disabled or future-facing pages may exist during the transition, but they must show a clear
  reason and a useful preview rather than inert decoration.

### 5.5 Layout rules

The UI should stay dense, operational, and readable.

- Use page-level workspaces, not nested card piles.
- Keep repeated items as cards, but avoid cards inside cards.
- Use compact headers for operational pages instead of hero-style marketing layouts.
- Keep the command-center identity, but reduce visual noise.
- Preserve stable dimensions for fixed-format controls, bracket slots, calendar cells, score panels,
  and compact stat tiles.
- Every page should be usable at common desktop widths and degrade cleanly on smaller screens.

## 6. Future-System Preparation

### 6.1 Larger games or events selection

`v0.2.4` should introduce a page that can eventually support around `32` available games/events.

The first pass can use local fictional event definitions with a shape like:

```ts
interface GameEventDefinition {
  id: string;
  name: string;
  category: "local" | "regional" | "elite" | "major";
  week: number;
  drawSize: 16 | 32;
  prestige: number;
  status: "available" | "locked" | "completed";
}
```

This page should focus on selection, comparison, and preview. It should not force the full tournament
system to become `32`-draw capable yet.

Preparation target:

```text
32 available competitions -> choose active event -> existing playable tournament loop
```

The existing 16-player knockout can remain the playable event format until the tournament reference
is updated.

### 6.2 Multi-game season system

The season page should establish the future campaign frame:

- season name or circuit label
- current week
- managed athlete record
- recent event results
- upcoming event queue
- progress toward a season goal

The first implementation can be a UI scaffold backed by simple local data. Full season scoring,
ranking points, player fatigue across events, and save migration should be separate work unless they
are explicitly scoped into the implementation pass.

The intended future loop is:

```text
choose event -> prepare match -> play tournament -> review result -> advance calendar -> choose next event
```

### 6.3 Calendar system

The calendar should help the player understand time and planning.

It should eventually show:

- event weeks
- match days
- travel or recovery windows
- completed results
- next playable item
- season milestones

For `v0.2.4`, a useful first pass is a readable month or week grid connected to event preview data.
The calendar may be non-simulating at first, but it should share the same event definitions as the
Games and Season pages so the UI does not fork into parallel fake systems.

### 6.4 Relationship to the current tournament

The current tournament remains the working game loop.

The new pages should frame it like this:

```text
Games Page
    |
    v
Season / Calendar Context
    |
    v
Current Event Bracket
    |
    v
Live Managed Match
    |
    v
Review
```

## 7. Data And State Boundaries

`v0.2.4` should separate UI navigation state from simulation state.

Recommended state split:

- `activePage`: presentation navigation only
- `activeOverlay`: modal or drawer state only
- `selectedPlayerId`, `plannedTacticKey`, `tournament`, and `liveMatch`: existing gameplay state
- future `seasonPlan` or `calendarEvents`: local structured data, not gameplay formulas inside UI

The core boundary remains:

$$
\text{React UI} \rightarrow \text{intent} \rightarrow \text{game engine}
\rightarrow \text{state + events}
$$

If season or calendar data becomes persisted, the save schema must be versioned and migrated in the
same release. If it is only static preview data, the existing save format should not be changed.

## 11. Risks And Mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Future pages feel fake because season logic is not ready | Medium | Make preview states honest and connect them to the current tournament loop where possible |
| UI refactor breaks the existing tournament flow | High | Convert current views into pages before adding new feature pages |
| Overlay system becomes another source of clutter | Medium | Use one typed overlay host and keep modal ownership centralized |
| Scope creeps into full season simulation | High | Treat season and calendar as scaffolds unless save schema and subsystem rules are explicitly updated |
| `32` event cards imply 32-player draws too early | Medium | Separate event selection size from tournament draw size until tournament rules change |
| Navigation becomes decorative | Medium | Every page must define a decision, context, action, and feedback loop |

## 12. Relationship To Other Versions

`v0.2.3` makes match simulation stronger.

`v0.2.4` makes the interface strong enough to hold more game structure.

`v0.3` can then expand the manager loop with fewer UI debts, because Games, Season, and Calendar
will already have homes in the product.

In short:

```text
v0.2.3 -> how matches work
v0.2.4 -> where features live
v0.3   -> how seasons progress
```
