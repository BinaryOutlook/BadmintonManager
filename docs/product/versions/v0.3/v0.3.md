# V0.3 Stable Career System

Status: Active / stable career-system foundation
Owner: BinaryOutlook
Last updated: 2026-05-18
Iteration type: Career loop stabilization, local save trust, and controlled manager expansion

## 1. Summary

`v0.3` is the point where **Badminton Manager** stops treating career play as scaffolding and starts
treating it as the stable long-run foundation.

The release introduces a coherent local-first career system:

```text
Start Career -> Career Command Center -> Training / Calendar / Match Planning
             -> Live Match -> Post-Match Review -> next scheduled decision
             -> Save Manager
```

The system is stable, but it is deliberately not maximal. The current career model proves the loop
with a locked managed athlete, fictional events, simplified rankings, bounded training/recovery,
budget pressure, scheduled match days, autonomous universe event completion, and safe local saves.

The version philosophy is:

$$
\text{Stable Career}
\neq
\text{Fully Realistic Career}
$$

`v0.3` earns the right to expand. Later versions should add more flexibility and realism in layers,
without destabilizing deterministic simulation, save compatibility, or the compact management UI.

## 2. Release Thesis

The player fantasy is now bigger than a disposable bracket run. The player can still launch a quick
tournament, but the primary long-run promise is a coach-managed career.

The career loop should feel like:

$$
\text{Career Value}
=
\text{clear next decision}
+ \text{visible consequences}
+ \text{safe persistence}
+ \text{sport-specific match reads}
$$

The interface should not pretend to be a huge sports universe yet. It should make the limited career
model trustworthy, readable, and easy to extend.

## 3. Stable In V0.3

The following surfaces are considered stable for the current release target:

- launch hub with resume, career start, quick tournament, Save Tools, Preferences, and recovery state
- locked managed-athlete career creation
- compact Career Command Center organized around the next decision, consequence summary, player condition, urgent tasks, calendar pressure, ranking/evidence, compact finance, and demoted ecosystem chips
- fictional event Calendar with entry gates, deadlines, scheduled match days, autonomous non-entered event outcomes, and active/past event views
- Training, Match Planning, Live Match, and Post-Match Review route flow
- unified Portal, 30-day Timeline, and month Calendar commitments sourced from one deterministic manager schedule
- Rally Pattern Map and compact live Match Command Center presentation
- local Save Manager with import preview, export, overwrite warning, active-slot deletion, and corrupt-save quarantine handling
- versioned and migrated local persistence, currently including `career.universeEvents`, `career.rankingResults`, preparation history, and save version `12`

## 4. Intentional Limits

These limits are product decisions, not missing polish:

- one primary managed athlete is the career anchor
- event tiers, rankings, and calendar logic are fictional and simplified
- training and recovery are bounded preparation choices, not a full development lab
- the program ecosystem now supports a small Lead/Rotation/Development roster, shared scheduled preparation,
  evidence-backed recruitment, weekly payroll, and deterministic daily talent-pool movement; negotiation, staff
  allocation, and intake/graduation lifecycles remain intentionally bounded
- economy is a lightweight pressure model, not a full accounting game
- no backend, auth, database, cloud save, real athlete likenesses, or licensed event content

The expansion path should be gradual:

```text
v0.3: stable career spine
v0.3.x: sharper feedback, tuning, and save hardening
v0.4+: deeper flexibility, realism, and system separation
```

## 5. Product Boundaries

`v0.3` must preserve:

- local-first operation
- deterministic match and universe outputs for identical seeds, players, tactics, calendar state, and choices
- simulation logic outside React components
- UI as presentation plus player intent
- versioned, validated persistence
- fictional content boundaries
- coach-not-athlete interaction

Do not add the following without a fresh decision:

- backend services
- auth or accounts
- database-backed saves
- direct racket controls
- licensed athlete or tournament likenesses
- a replacement UI framework or state library

## 6. Verification Expectations

Keep these commands green:

- `npm run typecheck`
- `npm run test`
- `npm run build`
- `npm run test:e2e`

The active `v0.3` proof should also keep focused coverage around:

- start-screen empty, active-save, mobile, and recovery states
- Career Command Center bounded desktop/mobile layout
- Calendar scheduling and due-match routing
- Save Manager import/export/delete/recovery behavior
- Rally Pattern Map and `Finish Set` match transport
- post-match continuation into later rounds and completed event states

## 7. Relationship To Earlier Versions

```text
v0.2.3 -> match simulation fidelity baseline
v0.2.4 -> UI framework, player profile, launch/save/career shell bridge
v0.3   -> stable career loop
```

`v0.2.4` remains important historical context, but it is no longer the active target. Its job was to
make the interface strong enough to hold career play. `v0.3` is where that career play becomes the
main foundation.

## 8. Open Expansion Questions

- How much athlete growth should ship before the player needs deeper training controls?
- Which career systems should become more flexible first: calendar entry, scouting, tactics, staff, facilities, or recruitment?
- How realistic should rankings and event qualification become before they risk overwhelming the player?
- Should post-`v0.3` navigation remain internal SPA state, or should URL routes become worthwhile?
- How much save migration coverage is enough before broader career-system refactors?
