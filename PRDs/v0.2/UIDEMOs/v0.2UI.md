# V0.2 UI Brief

Status: Draft
Owner: BinaryOutlook
Last updated: 2026-04-22
Iteration type: Frontend visual redesign and UX hardening pass

## 1. Summary

`V0.2 UI` is a presentation-focused iteration for **Badminton Manager**.

The current build is already playable:

- a player can choose a singles athlete
- lock in a pre-match tactic
- enter a 16-player knockout tournament
- simulate matches set by set
- use a between-set team talk
- advance through the bracket

The problem is not feature absence.

The problem is that the current interface still looks like an early functional prototype instead of a believable sports management product.

This brief exists to guide a specialized frontend UI and UX generator toward a stronger visual identity and better screen composition without changing the core game logic.

## 2. Product Framing

This is not an arcade badminton game.

This is not a direct-control sports title.

This is a badminton management simulation where the player acts as:

- coach
- tactician
- match reader
- tournament manager

The UI must reinforce that identity at all times.

The product should feel closer to:

- a coaching console
- a tournament command center
- a match analysis desk

than to:

- a flashy mobile sports app
- a neon esports dashboard
- a generic startup landing page

## 3. Design Goal

The UI should feel:

- modern
- solid
- premium
- data-dense
- sport-specific
- serious

The closest inspiration point is the feeling of a **Football Manager**-style management dashboard, but the result must not look like a literal imitation.

The desired mood is:

- dark, grounded surfaces
- strong visual hierarchy
- compact but readable information density
- clear emphasis on control, analysis, and consequence

## 4. Core Visual Direction

### 4.1 Background and surface treatment

The app should use **solid backgrounds and grounded panels**.

Requirements:

- prefer layered dark surfaces over glassmorphism
- use tonal separation rather than heavy transparency
- allow subtle texture, grain, or restrained gradients only if they support solidity
- avoid floating cards that feel detached from the product shell
- avoid giant blurred blobs or decorative light fog

The interface should feel anchored and built, not airy and decorative.

### 4.2 Color direction

Preferred base family:

- charcoal
- graphite
- slate
- deep blue-black
- restrained green or cyan accents

Requirements:

- high readability
- strong but controlled contrast
- avoid pure black and pure white where possible
- avoid purple-heavy palettes
- avoid toy-like colorfulness

Accent usage should feel intentional and functional:

- selected state
- important action
- live status
- momentum or pressure cues

### 4.3 Typography

Typography should feel:

- editorial
- sharp
- deliberate
- premium

Requirements:

- clear hierarchy between title, section label, panel title, and data text
- avoid playful or overfriendly type choices
- avoid oversized landing-page hero styling inside operational screens
- dense views must remain highly scannable

### 4.4 Shape language

Requirements:

- panels should feel structured and substantial
- corners can be rounded, but not soft and bubbly
- borders should be visible enough to organize the page
- shadow should support depth, not dominate the look

The result should look like a serious analysis product, not a consumer toy.

## 5. Hard Style Guardrails

Do not generate:

- generic AI gradient-blob backgrounds
- excessive glassmorphism
- oversized empty spacing
- startup hero-page aesthetics
- purple-on-black cyberpunk styling
- playful sports-betting app vibes
- dashboard cards that all look identical and interchangeable

Do generate:

- strong shell structure
- solid panels
- clear density
- visual hierarchy that rewards scanning
- restrained but confident accents

## 6. UX Goals

The redesign should make the product better at four things:

### 6.1 Readability

The player should immediately understand:

- where they are in the tournament
- who they are coaching
- what tactic is active
- what just happened in the match
- what action is available next

### 6.2 Match drama

The live match screen should feel important.

The player should feel:

- tension between sets
- momentum in the scoreline
- pressure in the commentary feed
- consequence in each coaching intervention

### 6.3 Product identity

The app should visually communicate:

- coaching
- management
- analysis
- tournament progression

It should not visually drift toward:

- action gameplay
- fantasy RPG systems
- casual sports minigames

### 6.4 Future expandability

The UI should leave room for future systems such as:

- deeper tactics
- more stats
- player development
- season progression
- scouting and match analysis

That means layout and components should not be so brittle that every future system requires a full redesign.

## 7. Technical Constraints

The UI generator must respect the current architecture.

Requirements:

- stack remains `React + TypeScript + Vite`
- preserve the current game loop and store behavior
- do not move gameplay logic into UI components
- do not rewrite the simulation engine
- do not change how the deterministic core resolves matches
- avoid introducing heavy UI frameworks unless absolutely necessary
- prefer CSS variables or theme tokens for reusable design decisions

The correct work is a **presentation redesign**, not an architecture rewrite.

## 8. Required Screens

The redesign must include the following screens and states.

### 8.1 Global app shell

This is the persistent framing layer around the active screen.

It should include:

- a top-level identity treatment for Badminton Manager
- current player identity
- current phase or mode
- visual continuity across the product

It should not dominate the screen more than the active gameplay surface.

### 8.2 Setup screen

Current purpose:

- choose the managed player
- inspect a compact identity summary
- choose the opening tactical lens
- start the tournament

This screen should feel like:

- tournament registration
- athlete selection
- tactical briefing

It should contain:

- selected player feature panel
- player pool browsing surface
- tactic selection surface
- clear primary action to start the run

Desired feeling:

- analytical
- anticipatory
- slightly ceremonial

### 8.3 Tournament overview screen

Current purpose:

- show the player’s current round
- show the next opponent
- let the player refine the tactic lock-in
- display the tournament path
- start the managed match

This screen should feel like:

- a tournament control room
- a knockout bracket desk
- a scouting and preparation board

It should contain:

- round and tournament context
- next opponent panel
- tactic lock-in panel
- bracket visualization
- action area for starting the match

Desired feeling:

- clear
- strategic
- organized
- competitive

### 8.4 Live match screen

This is the most important screen in the redesign.

Current purpose:

- show the score
- show set progression
- show stamina
- show commentary
- allow between-set team talks
- advance set by set

This screen should feel like:

- a live coaching console
- a match command center
- a broadcast desk crossed with a tactics room

It must contain:

- dominant scoreboard
- player identity and style context
- set history
- stamina read
- strong commentary feed
- team talk area
- clear next action

Desired feeling:

- tense
- premium
- controlled
- readable under pressure

### 8.5 Run complete screen

Current purpose:

- show whether the run ended in victory or elimination
- allow the player to review the bracket state
- let the player start again

This screen should feel like:

- a tournament recap
- a season-end snapshot
- a clean reset point

It should contain:

- outcome headline
- concise recap language
- bracket summary or path review
- clear CTA for a new run

Desired feeling:

- reflective
- conclusive
- earned

## 9. Required Panels And Elements

The UI generator should ensure these elements are designed intentionally rather than left as generic cards.

### 9.1 Hero or identity strip

Needs:

- project identity
- current player
- current mode
- subtle but strong shell framing

### 9.2 Player identity cards

Needs:

- player name
- nationality
- seed
- style label
- derived strengths

These should feel like athlete dossiers, not plain list rows.

### 9.3 Tactic selection cards

Needs:

- tactic name
- one-line intent
- tempo and risk cues
- strong selected state

These should feel like coaching stances, not generic settings toggles.

### 9.4 Bracket blocks

Needs:

- round grouping
- clear winner indication
- visual distinction for managed match
- readable progression

These should feel like a tournament structure, not just stacked boxes.

### 9.5 Scoreboard

Needs:

- strong match importance
- immediate recognition of set lead
- readable player labels
- room for supporting context

This should be the visual anchor of the live match screen.

### 9.6 Commentary feed

Needs:

- hierarchy between recent and older items
- compact but readable event cards or rows
- clear labeling by set or score state
- emotional importance

This feed must not feel like an afterthought.

### 9.7 Team talk controls

Needs:

- clear moment of intervention
- strong distinction from passive information panels
- concise descriptions
- visible consequence framing

### 9.8 Completion recap

Needs:

- emotionally satisfying finish state
- visible bracket outcome
- clear reset path

## 10. Screen-Level Layout Guidance

### 10.1 Setup screen layout

Recommended hierarchy:

1. screen header and purpose
2. selected player focus panel
3. tactic decision area
4. player pool browsing
5. primary CTA

The player should understand the athlete and the tactical choice before scanning the full pool.

### 10.2 Tournament overview layout

Recommended hierarchy:

1. round and status header
2. next opponent and tactic decision zone
3. bracket view
4. primary CTA to start the managed match

The screen should emphasize readiness and path.

### 10.3 Live match layout

Recommended hierarchy:

1. scoreboard and match identity
2. live contextual data such as stamina and set history
3. commentary feed
4. intervention area for team talks
5. primary next action

The live match screen should feel denser and more intense than the other screens.

### 10.4 Completion layout

Recommended hierarchy:

1. result headline
2. short recap
3. tournament path review
4. reset action

## 11. States To Design For

The generator should account for more than the happy path.

Required states:

- default setup state
- selected player state
- selected tactic state
- tournament overview with pending managed match
- live match with no completed sets yet
- live match with at least one completed set
- live match with team talk choices active
- live match completed state
- run complete with championship result
- run complete with elimination result

Support states if needed:

- hover
- selected
- disabled
- scroll-heavy commentary feed
- narrow desktop and mobile layouts

## 12. Responsiveness

The product is desktop-first, but it still needs to work on smaller screens.

Requirements:

- desktop should feel like the primary experience
- tablet and narrow laptop states should preserve hierarchy
- mobile should remain readable and functional
- critical actions must stay obvious across breakpoints

Do not collapse the interface into a generic mobile card stack without hierarchy.

## 13. Motion And Interaction

Motion should be present, but restrained.

Good motion:

- panel entrance or screen arrival transitions
- selected state emphasis
- feed updates that feel alive
- subtle scoreboard emphasis

Bad motion:

- constant bouncing
- decorative floating
- flashy parallax
- motion that makes dense screens harder to read

The app should feel responsive and premium, not animated for animation’s sake.

## 14. Deliverables Expected From The UI Generator

The frontend generator should produce:

- updated screen layouts for all required screens
- cohesive visual styling across the app
- improved panel composition and hierarchy
- reusable design tokens or CSS variables
- polished states for selection, match flow, and completion

It may also produce:

- small presentational helper components
- better composition of current React components
- more deliberate spacing and typography systems

It should not produce:

- gameplay rewrites
- new simulation rules
- hidden assumptions that require backend work

## 15. Success Criteria

`V0.2 UI` is successful when:

- the app feels like a credible sports management product
- the live match screen feels dramatic and important
- the product identity reads as coaching and tournament management
- the interface uses solid, modern backgrounds and grounded panels
- the other AI generator improves the visuals without breaking logic
- future systems can be added without the layout feeling trapped

## 16. Handoff Note For The Specialized UI Generator

Treat this as a **UI and UX redesign brief for an existing working app**.

Your job is to:

- elevate the visual identity
- improve clarity
- improve hierarchy
- make the app feel premium and intentional

Your job is not to:

- invent a different game
- turn it into an action-sports product
- replace the simulation with flashy presentation

The project should come out of this iteration looking like a stronger version of itself, not like a completely different genre.
